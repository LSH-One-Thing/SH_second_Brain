> [!important] 핵심 Takeaway (콘텐츠 방향성 관점)
> AI가 코딩을 대신하는 시대, "엄밀함"은 사라진 게 아니라 위치를 바꿨다. **코드 작성 → 설정/보안 → 시스템(하네스) → 문제 정의** 순으로 이동 중이다. 비개발자가 해커톤을 이기는 이유, 바이브코딩 서비스가 5분 만에 뚫리는 이유, 하네스 엔지니어링이 등장한 이유가 이 하나의 원리로 설명된다. 진짜 경쟁력은 코딩 실력이 아니라 **"어떤 문제를 풀 것인가"를 가장 정확하게 아는 사람**에게 이동했다.

## 엄밀함의 이동

Chad Fowler의 "Relocating Rigor" — **엄밀함은 사라지지 않는다, 더 높은 추상화 수준으로 이동할 뿐이다.**

XP 운동이 설계 문서 → 테스트 코드로, 동적 언어가 컴파일러 검사 → 런타임 테스트로 엄밀함을 이동시켰듯이. 매번 기존 진영은 "그건 엄밀함을 포기하는 것이다"라고 했고, 매번 틀렸다.

AI 개발 패러다임은 2022-2026년 사이 세 번 바뀌었다:

| 시대 | 기간 | 질문 | 한계 |
|---|---|---|---|
| 프롬프트 엔지니어링 | 2022-2024 | 어떻게 **말할** 것인가 | 같은 입력에 다른 출력. 비결정론적 |
| 컨텍스트 엔지니어링 | 2025 | 무엇을 **넣을** 것인가 | 정보가 너무 많으면 AI가 대충 읽는다 |
| 하네스 엔지니어링 | 2026 | 어떤 **시스템을 만들** 것인가 | — |

각 전환의 동인은 이전 패러다임이 약속한 것을 지키지 못했기 때문이다. 엄밀함이 빠진 자리에서 실패가 일어난다.

---

### 바이브코딩의 숙취: 설정에서 엄밀함이 빠지면

> "친구가 바이브코딩으로 만든 서비스, 5분 만에 뚫었습니다. URL 하나 쳐봤을 뿐인데 관리자 페이지가 그냥 열렸습니다."

OWASP가 2021년부터 취약점 1위로 꼽는 것: **접근 권한 설정 실수.** 코드를 못 짜서가 아니라 문을 안 잠가서 뚫린다. AI는 "이 페이지는 관리자만 볼 수 있게 해줘"라고 시키지 않으면 하지 않는다.

**바이브코딩 서비스 보안 체크리스트 5가지:**

1. **어드민 페이지 잠금** — `/admin` URL로 바로 접근되면 공격 시작점. 인증 미들웨어 필수. AI에게: *"어드민 라우트에 인증 미들웨어 걸어줘"*
2. **Supabase RLS 설정** — Row Level Security 없으면 API 호출만으로 전체 이메일·결제내역 조회 가능. 현관문 잠금 여부. 개인정보보호법 위반 직행
3. **환경변수 유출 차단** — F12 개발자 도구 → 소스/네트워크/콘솔 탭 확인. API 키가 브라우저에 노출되면 금고 비밀번호 공개 수준
4. **결제 금액 서버 검증** — 프론트엔드에서 넘어오는 가격을 서버가 그대로 믿으면 100원에 상품 판매 가능. 서버에서 DB 값으로 검증 필수
5. **에러 메시지 필터링** — DB 테이블명·파일 경로·스택트레이스가 브라우저에 노출되면 도둑에게 집 구조도 제공. 프로덕션에서는 상세 에러 숨기기

==보안 구멍은 코드가 아니라 설정에 있다.== 기능 생성은 AI가 했지만, 보안 설정의 엄밀함을 아무도 챙기지 않은 결과다.

### 하네스 엔지니어링: 엄밀함을 시스템으로 강제하다

> "프롬프트로 '규칙 좀 제발 지켜줘'라고 애원하는 게 아니라, 룰을 어기면 시스템이 스스로 제동을 걸고 에러를 뱉어 다시 수정·검증하게 만드는 자동화 무한 루프 시스템."

AI 개발의 병목은 **인간**이었다. AI가 코드를 만드는 속도는 빠른데, 인간이 하나하나 테스트하다 보니 생산성의 병목이 생겼다. OpenAI 팀의 결론: 인간을 삭제하자. AI가 인간의 개입 없이 최대한 스스로 만들 수 있게 하는 시스템.

인간 행동심리학과 동일한 구조 — 의지가 아닌 환경을 바꾸는 것:

| 인간의 자기통제 | 하네스 엔지니어링 |
|---|---|
| 다이어트 중 배달 앱 삭제 | Git 워크트리로 메인 코드 분리 |
| 매일 눈바디 사진 기록 | AI 실행 과정 로그·스크린샷 |
| 벌금 스터디 강제 규칙 | 계획서 없이 커밋 시도 → Husky 차단 |

**하네스의 3요소:**

**① 로그 시스템 (가시성)** — AI에게 눈을 달아준다. 모든 실행 결과를 기록하여 AI가 스스로 피드백하게. 로그가 있어야 AI가 "내가 실수했네"를 인식하고 수정한다.

**② 컨텍스트 목차화 (점진적 공개)** — 하나의 큰 파일에 모든 걸 때려넣으면 AI가 앞부분만 읽고 뒤는 대충 추측한다. 목차 파일 → 필요할 때 해당 문서를 꺼내는 구조. 컨텍스트가 많을수록 지침이 지침이 되지 않는다.

**③ CI/CD 강제 (자동 제동)** — ESLint 룰, Husky 프리커밋 훅, 빌드·린트·단위 테스트 자동화. 어기면 에러, 통과해야 커밋. 말로만 "이렇게 해줘"가 아니라 어겼을 때 잡아내는 장치.

**하네스가 있으면 자동화되는 5단계:**
계획 문서 작성 → 워크트리에서 구현 → 단위 테스트 작성 → 전체 검증(린트·빌드·파일 크기) → 커밋·머지·완료 처리

> [!tip] 도입 시점
> 모든 것을 한 번에 구축하려 하지 말 것. 반복 작업이 느껴질 때마다 하나씩 자동화하다 보면 나만의 하네스가 만들어진다.

### 비개발자가 이기는 이유: 엄밀함이 문제 정의로 이동

13,000명 지원 · 경쟁률 26:1 · Claude Code 해커톤.

| 순위 | 직업 | 프로젝트 | 해결한 문제 |
|---|---|---|---|
| 1등 | 캘리포니아 건설업자 | CrossBeam | 건축 허가 반려 처리 몇 주 → 15분 |
| 3등 | 심장내과 의사 | postvisit.ai | 진료 기록 → 환자가 이해하는 언어로 |
| 특별상 | 우간다 도로기술자 | Tara | 블랙박스 영상 → 인프라 투자 보고서 자동 생성 |

공통점: **자기 분야의 문제를 누구보다 깊이 이해하고 있었다.**

심사 기준은 코드의 우아함이 아니라 문제 해결력이었다. 건설업자는 매일 겪는 허가 지옥을 알았고, 의사는 환자가 진료실을 나서면 뭘 모르는지 정확히 알았다. 클로드 코드가 코딩 실력의 간극을 메웠고, 남은 건 순수한 **문제 정의 능력**이었다.

> "기술을 가진 사람이 문제를 찾던 시대에서, 문제를 가진 사람이 기술을 쓰는 시대로."

---

### PRD: 엄밀함을 가장 상류에 배치하다

바이브코딩의 한계 — 수정 요청 시 멀쩡한 부분이 망가지는 "스파게티 코드" — 는 **초반 규격 없이 진행하기 때문**이다. 하네스 시스템이 완벽하게 작동하려면 초반 아키텍처 구성이 엄청나게 디테일해야 한다. 이것이 PRD(Product Requirement Document, 제품 요청서)의 역할이다.

**AI와 함께하는 PRD 작성 프로세스:**

```
1. 문제 정의
   "코딩 학습 앱" (X)
   "즉각적인 성취 경험 부재로 인한 포기 문제 해결" (O)
   → 페이스북·링크드인·틱톡은 모두 소셜미디어지만 해결하는 문제가 달랐다

2. AI 페르소나 시장조사
   "너는 22살 대학생이야, 코딩을 배우려다 포기했던 날 얘기해줘"
   → 10분에 50명 인터뷰 효과

3. 패턴 찾기
   "방금 한 모든 유저 인터뷰에서 공통 패턴이 뭐야?"
   → 즉각적인 좌절 + 성취감 부재

4. 핵심 가치 하나만
   "단 하나만 해결한다면?"
   → 단일 핵심 기능으로 집중

5. PRD 작성 (4가지 포함)
   - 이 문제를 해결하는 핵심 기능 딱 하나
   - 이게 효과 있다는 걸 증명할 성공 지표
   - 우리가 명시적으로 만들지 않을 것
   - 기술적 제약사항

6. 실패 가능성 묻기
   "이 아이디어가 실패한다면 이유가 뭐야?"

7. 암묵적 가정 검증
   "이 프로젝트에서 우리가 어떤 가정을 하고 있나요?"

8. 엣지 케이스 발굴
   생각지도 못한 예외 상황을 미리 발굴
```

---

### 통합 구조: 엄밀함이 이동하는 순서

```
[문제 정의 · PRD]  →  [컨텍스트 설계]  →  [하네스 구축]  →  [바이브코딩]  →  [보안 체크]
       ↑                     ↑                   ↑                 ↑                ↑
  도메인 전문성           목차화                CI/CD             기능 구현         설정 엄밀함
  (비개발자의 힘)         점진적 공개            훅·워크트리         (AI가 담당)       (5가지 체크)
```

**엄밀함을 생략하면 생기는 일:**

- PRD 없이 → 수정 요청마다 멀쩡한 부분 망가짐 (스파게티 코드)
- 하네스 없이 → AI가 규칙 무시, 인간이 매번 테스트하며 병목
- 보안 설정 없이 → 5분 만에 서비스 뚫림
- 문제 정의 없이 → 기술 있는 개발자가 문제 있는 비개발자에게 패배

==엄밀함은 사라지지 않았다. 코드에서 설정으로, 설정에서 시스템으로, 시스템에서 문제 정의로 이동했을 뿐이다.==
